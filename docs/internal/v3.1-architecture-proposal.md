**Tek temiz disiplin:** proje kökünde `AGENTS.md` (AI discovery için) + `.kortext/` (her şey içeride, `.git/` gibi).

### v3.0'da kaybedilen şey

v3.0 paketleme turunda (Faz 10) "npm package convention" gerekçesiyle yapı düzleştirildi:

```
proje/
├── AGENTS.md
├── agents/                  ← proje köküne çıktı, gereksiz kopya
├── workflows/               ← aynı
├── rules/                   ← aynı
├── workspace/               ← projeye özgü ama .kortext'ten ayrı
│   ├── references/
│   ├── reports/
│   └── memory/
│       ├── context/         ← markdown'da ama engine state olmalı
│       ├── backlog/         ← v2 artığı, SQL'e taşındı ama dosya hâlâ duruyor
│       ├── decisions.md
│       ├── handover.md
│       └── learned.md
└── .kortext/                ← sadece engine state
```

Sonuç:
- Proje kökü kalabalık (6 dizin + AGENTS.md)
- `agents/ workflows/ rules/` her `kortext init`'te tekrar tekrar diske düşüyor (gereksiz)
- Kortext upgrade'i otomatik yayılmıyor (her proje init zamanı snapshot'unda donuyor)
- `workspace/memory/context/` markdown'da ama paralel ajan çakışma riski (SQL transaction yok)

### v3.1.0'ın yapacağı

v2 disiplinine geri dön + v3'ün SQL state katmanını koru + bugünkü tartışmadan çıkan disiplinleri ekle:
- Per-file rapor (her rapor ayrı dosya, monolitik şişme yok)
- Frontmatter standartlaştırma (4 ayrı net standart: references/reports/handover/ADR+learned)
- ADR pattern (tek `decisions.md`, TOC'lu, mevcut iskelet korunur)
- Handover rotation (5 devirde `handover-<YYYY-MM-DD-HHMM>.md`)
- TOC + selective read pattern (büyük dosyalar için)
- Prompt cache aktivasyonu (`--system-prompt` flag kontrolü ve aktivasyon)
- Workflow/persona için SQL index (parse-time validation)
- Backlog UI form ("+ New Item")
- Dashboard global readonly editor

## 2. Karar prensibi: Markdown vs SQL

Verinin nereye konacağı kararı **değişim sıklığı değil**, **kullanım biçimine göre verilir.**

> **Markdown = "okumak / düşünmek için."  SQL = "saymak / aramak / sıralamak için."**

Üç soru:

1. **AI bunu doğrudan okuyup ona göre davranacak mı?** → Markdown
2. **Dashboard'da filtrelenecek, sıralanacak, sayılacak mı?** → SQL
3. **İki ajan/kullanıcı aynı anda değiştirebilir mi?** → SQL (transaction)

Hibrit (`backlog_items` gibi) çift sorulara cevap verir: structured alanlar SQL kolon (`status, owner, parent_id`), gövde markdown TEXT (`body_md`).

## 5. Frontmatter standartları (final)

**4 ayrı standart** — dosyanın **niteliği** (kanonik kaynak vs kayıt akışı vs devir notu vs ADR/ders) standardı belirler.

### 5.1 References — file-level, kanonik kaynak

Sabit dosya, neredeyse hiç güncellenmez. Reject olursa baştan yazılır. Approver onayı zorunlu.

```yaml
---
status: uninitialized | writing | waiting | approved | rejected
author: +<persona>           # zorunlu
reviewer: +<persona>         # opsiyonel
approver: +<persona>         # zorunlu
---
```

`updated_at` yok — neredeyse statik. `waiting` references'a özgü status (approver onayı bekliyor).

### 5.2 Reports — file-level, durum kaydı

Sabit kayıt, durum açıklar, yoruma gerek yok. Onay bekleyen bulgu varsa task açılır, task üzerinden onay alınır.

```yaml
---
status: uninitialized | writing | approved
author: +<persona>           # zorunlu
reviewer: +<persona>         # opsiyonel
updated_at: 2026-05-24T16:42:00Z
---
```

`approver` yok — rapor onay gerektirmez.

### 5.3 Handover devir kaydı — entry-level

Her devir kaydı **kendi frontmatter'ı ile başlar** (file-level değil entry-level). Approver/reviewer yok — devir notu, onaya tabi değil, sadece "ne yaptığını" yazıyor. `to:` alanı **opsiyonel** (sıradaki kim alırsa o devralır).

```yaml
---
status: uninitialized | writing | approved
author: +<persona>           # zorunlu
updated_at: 2026-05-24T16:42:00Z
---
```

Dosyada N devir kaydı = N adet ayrı frontmatter bloğu (her biri kendi `## Handover: <task-id>` bölümünü başlatır).

### 5.4 ADR (decisions.md) ve Knowledge (learned.md) — mevcut iskelet korunur

`_docbase/kortext/workspace/memory/decisions.md` ve `learned.md`'deki **mevcut iskelet aynen korunur**. Her ADR/öğrenim kaydı, dosya içinde inline header pattern ile başlar:

**ADR pattern:**
```markdown
## ADR-[ID]: [ADR Title]
**Author:** [+ajan] | **Status:** [Waiting | Approved/Rejected | YYYY-MM-DD-HH:MM] | **Approver:** [+ajan]

### Context
### Decision
### Consequences
```

**Learned pattern:**
```markdown
## Öğrenim: [Hata/Problem Başlığı]
**Author:** [+ajan] | **Status:** [Waiting | Approved/Rejected | YYYY-MM-DD-HH:MM] | **Approver:** [+ajan]

### Problem
### Root Cause
### Solution
### Prevention
```

**Eklenen:** Dosyanın başına **markdown TOC** (Bölüm 7). **Engine** her yeni ADR/learned eklendiğinde TOC'a satır ekler (otomasyon engine sorumluluğunda — `+operation-manager` maintenance turuna bırakılmadı).

### 5.5 INFO callout kaldırıldı

Şu an her references/reports dosyasında `> [!INFO]` callout aynı bilgiyi tekrar ediyordu (status, author). Bu kalkar. **Tek kaynak frontmatter** — dashboard render eder.

## 6. File naming patterns

### Backlog (template'ler)

`templates/backlogs/` altında 6 prefix iskelet:

| Prefix | Tip | SQL `type` enum |
|---|---|---|
| `TXX-[task-name].md` | Task | `task` |
| `BXX-[bug-name].md` | Bug | `bug` |
| `EXX-[epic-name].md` | Epic | `epic` |
| `DXX-[debt-name].md` | Debt | `debt` |
| `SXX-[spike-name].md` | Spike | `spike` |
| `HXX-[hotfix-name].md` | Hotfix | `hotfix` |

Engine "yeni item aç" deyince template'ten okur, frontmatter parse → SQL kolonlarına + JSON, body_md kolonuna gövde. Item dosyaya yazılmaz; sadece SQL.

ID format: `T01`, `T02`, `B01`, `E01-auth`, … (XX yerine sırasal numara veya kısa slug).

### Reports (per-file)

```
<scope>_<slug>_<YYYY-MM-DD-HHMM>.md
```

Örnekler:
```
reports/test-reports_login-flow_2026-05-24-1432.md
reports/test-reports_payment-edge-cases_2026-05-25-0918.md
reports/security-reports_auth-module-audit_2026-05-26-1100.md
reports/delivery-reports_v1.2.0-staging_2026-06-01-1530.md
reports/release-notes_v1.2.0_2026-06-02-0900.md
```

- `<scope>` = kategori (eski tek-dosya adı — `test-reports`, `security-reports`, …)
- `<slug>` = neyi raporladığı (kebab-case)
- `<timestamp>` = ISO uyumlu, sortable

Engine yeni rapor yazıldığında **dosya yaratır**, SQL `reports_index` tablosuna meta yazar:
```
id, scope, slug, file_path, author, status, tags (JSON array), created_at, related_item
```

### Handover rotation

```
.kortext/memory/handover.md                              ← canlı (latest)
.kortext/memory/handover-2026-05-16-1430.md              ← ilk rotation
.kortext/memory/handover-2026-05-22-0912.md              ← ikinci rotation
...
```

Rotation kuralı: **5 devir VEYA dosya > 30KB** (hangisi önce). `+operation-manager` `09-maintenance-cycle`'da kontrol eder. Manuel `kortext archive handover` komutu da olsun.

Naming pattern: `handover-<YYYY-MM-DD-HHMM>.md`
- `YYYY-MM-DD-HHMM` = arşivin **son devrinin** timestamp'i

Sırasal numara yok — timestamp tek başına sortable + kapsadığı tarih dosya adından okunur.

Regex parse: `^handover-(\d{4}-\d{2}-\d{2}-\d{4})\.md$`

### ADR (`decisions.md` tek dosya — mevcut iskelet + TOC)

Eray "her ADR ayrı dosya gereksiz, sade tek dosya tut" kararı. **Mevcut iskelet aynen korunur** (Bölüm 5.4). Üstüne TOC eklenir:

```markdown
# Architectural Decision Records (ADR)

## İçindekiler

1. [ADR-001 — Auth Stack: Auth0](#adr-001-auth-stack-auth0)
2. [ADR-002 — Payment Provider: Stripe](#adr-002-payment-provider-stripe)
3. [ADR-003 — DB Migration Approach: Prisma Migrate](#adr-003-db-migration-approach-prisma-migrate)

---

## ADR-001: Auth Stack: Auth0
**Author:** +engineering-manager | **Status:** Approved | 2026-05-10-14:30 | **Approver:** +prime

### Context
...

### Decision
Auth0 kullanılacak.

### Consequences
- ✅ Hızlı entegrasyon
- ⚠️ Aylık $X maliyet eşiğinde

---

```

SQL `decisions_index` her ADR'yi satır olarak izler: `decision_id, title, status, markdown_path, anchor, item_id, decided_at`. `markdown_path` hep `memory/decisions.md`, anchor `#adr-001-auth-stack-auth0`. Dashboard tıklama anchor'a scroll.

### Learned (`learned.md` tek dosya, asla arşivlenmez — mevcut iskelet + TOC)

Aynı pattern. **Mevcut Knowledge Base iskeleti** (Problem/Root Cause/Solution/Prevention) korunur, üstüne TOC:

```markdown

### Solution
...
```

## 7. TOC + selective read pattern

### TOC (zorunlu, engine otomatik üretir)

Memory dosyaları (`learned.md`, `decisions.md`) ve büyük referans dosyaları başında **markdown standart TOC** olsun:

```markdown
## İçindekiler

1. [Bölüm Başlığı](#bölüm-başlığı)
2. ...
```

Anchor link'ler markdown native (GitHub, dashboard DOMPurify render'da çalışır). **Engine sorumluluğunda:** her yeni ADR/learned eklendiğinde TOC bölümüne satır eklenir (parse + insert).

### Selective read (v3.1.0 scope — workflow .md disiplini)

Claude Code Read tool `offset/limit` parametreleri destekliyor. Büyük dosyalar için pattern:

```
1. Ajan: Read(file, offset=0, limit=30)         ← sadece TOC
2. Ajan: Grep("^## Bölüm", file)                ← satır numarası bul
3. Ajan: Read(file, offset=N, limit=M)          ← sadece o bölümü oku
```

Verim: 30KB dosya tam yükleme = ~7500 token; TOC + bir bölüm = ~1500 token. **5x token tasarrufu** + prompt cache invalidation azalır.

Bu pattern workflow .md'lerinde her büyük dosyaya referans veren step'in talimatına yazılır: *"Bu dosya >30KB ise önce TOC oku, sonra ihtiyacın olan bölümü selective oku."*

## 9. Token mekaniği ve prompt cache

### Ajan'a giden prompt yükü (typical step)

| Parça | Token | Niye |
|---|---|---|
| Step talimatı (workflow'tan extract) | ~150 | "Şu dosyayı oku, şu çıktıyı yaz" |
| Persona tanımı (kendisi) | ~1250 | "Sen kimsin, sorumlulukların" |
| İlgili rule (örn. branching) | ~500 | Sadece ilgili |
| Referans dosyalar (blueprint, tech-stack…) | 1250-7500 | İşin asıl malzemesi |
| Önceki handover | ~500 | Devir notu |
| Kod base okuma (worktree) | 0-50000 | Asıl ağır iş |

Çekirdek workflow+persona+rule ≈ 2000 token; modern Claude/AGY pencere 200K-1M. **Çekirdek %0.2-1, gürültü düzeyinde.** Asıl yük referans dosyalarda ve kod okumada.

### v3.1.0 kazanımları

1. **Engine "bir kez parse, küçük talimat ver"** — ajan tüm workflow.md'yi görmez
2. **Prompt caching** — aynı persona/workflow tekrarlı kullanımda ~%90 indirim. **Kritik:** `claude-cli-executor.ts`, `antigravity-cli-executor.ts`, codex executor'da `--system-prompt` flag aktif olmalı + Claude API `cache_control` kullanılmalı. Şu an aktif değil — aktive edilecek (TODO listesinde Kritik #2).
3. **Selective read** — büyük dosyalar için TOC + offset (Bölüm 7)
4. **Workflow/persona için SQL index** — parse-time validation (`+ajan` placeholder kaçakları reddedilir); dashboard cross-cut sorgular ("persona iş yükü", "hangi workflow hangi referansı kullanıyor")

## 11. Eklenen kavramlar

### `templates/` (init seed)

`agents/ workflows/ rules/`'ten farklı: bunlar runtime yasası, **çoğaltılmaz** (global'den okunur). `templates/` ise sadece `kortext init` zamanında **bir kez kopyalanan** başlangıç dosyaları. Kopya sonrası bağ kesilir.

İçeriği iskelet olmalı (boş file değil) — frontmatter + bölüm başlıkları hazır, persona tutarlı format yazar.

### Per-file rapor

Monolitik `test-reports.md` → her test için `test-reports_<slug>_<ts>.md`. Avantajlar:
- Doğal segment (archive gereksiz)
- Paralel ajan çakışması yok (her ajan kendi dosyasını yazar)
- Dashboard SQL'den sıralı listele + filtrele
- Tags array desteği (`reports_index.tags` JSON kolonu) — ileride kategorizasyon zenginleşebilir

### Handover rotation

5 devir VEYA 30KB → `handover-<YYYY-MM-DD-HHMM>.md`. Latest `handover.md` daima taze. Geçmiş aynı klasörde, dosya adından kapsadığı tarih okunur.

### TOC + selective read

Büyük memory dosyaları (`learned.md`, `decisions.md`) başında markdown TOC. Engine TOC'u otomatik günceller (her yeni ADR/learned eklendiğinde). Ajan tam yükleme yerine TOC görüp Read(offset/limit) ile sadece ihtiyacı olan bölümü çekebilir.

### `kortext/` vs `.kortext/` ayrımı

Global pakette dot YOK (`kortext/` — npm convention); projede dot VAR (`.kortext/` — gizli framework klasörü). Aynı isim iki yer için kullanılmaz.

### Workflow/persona SQL index (yeni)

`workflow_steps` ve `personas` tabloları — markdown source kalır, engine boot'ta parse edip SQL'e upsert eder. Parse-time validation (`+ajan` placeholder reddedilir). Dashboard cross-cut sorgular.

### Backlog UI form ("+ New Item")

Dashboard Board ekranında modal — type/title/epic/owner/AC/dependencies girişi. REST `POST /api/backlog` → engine `add_backlog_item`. Template'ten `body_md` seed.

### Dashboard global readonly editor

Settings/Agents, Workflows, Rules sayfaları **readonly** — paket içi `.md` dosyaları render edilir. Yazma yok (paket immutable). İleride v3.2'de yazma mekanizması düşünülebilir; v3.1.0'da sadece görüntüleme.

### Prompt cache aktivasyonu

Tüm executor'larda (`claude-cli-executor`, `antigravity-cli-executor`, codex) `--system-prompt` flag + Claude API `cache_control` aktif edilir. Test edilir (aynı task 3x token usage karşılaştırması). Release notuna eklenir.

## 14. Strateji: Paralel worktree disipline'ı

v3.1.0 scope geniş. Tek branch'te lineer ilerlemek bottleneck yaratır. Çözüm: Kortext'in zaten kullandığı **git worktree isolation** disipline'iyle alt-fazları paralel sürmek.

Önerilen alt-faz dağılımı (her biri kendi worktree'sinde):

| Alt-faz | İş | Worktree |
|---|---|---|
| **Faz 11.4** | Faz 11.3 fix'leri commit + push (hazır iş) | `kortext/wt-1-phase-11-3-commit` |
| **Faz 12.1** | `workspace/` → `.kortext/` path refactor + engine routes | `kortext/wt-2-encapsulation` |
| **Faz 12.2** | `agents/workflows/rules` global'den okuma + init refactor | `kortext/wt-3-global-runtime` |
| **Faz 12.3** | `templates/` klasörü oluşturma + iskeletleri taşıma | `kortext/wt-4-templates` |
| **Faz 12.4** | Frontmatter standartlaştırma (mevcut .md'lerde) | `kortext/wt-5-frontmatter` |
| **Faz 12.5** | Per-file rapor mekaniği + `reports_index` tablosu | `kortext/wt-6-per-file-reports` |
| **Faz 12.6** | Handover rotation + TOC otomasyon | `kortext/wt-7-rotation-toc` |
| **Faz 12.7** | Prompt cache aktivasyon + selective read disipline | `kortext/wt-8-cache-selective` |
| **Faz 12.8** | Workflow/persona SQL index | `kortext/wt-9-sql-index` |
| **Faz 12.9** | Backlog UI form + global readonly editor | `kortext/wt-10-ui` |
| **Faz 13** | Workflow content rewrite (yeni mimari ile uyumlu) | `kortext/wt-11-workflow-content` |

Her worktree bağımsız test edilebilir, sonra ana branch'e merge. Kortext'in HANDOVER'daki #10 numaralı tasarım kararı (`kortext/run-<id>` worktree namespace) bu pattern için zaten hazır.

**Önemli not:** Faz 13 (workflow content rewrite) **diğer fazlardan sonra** merge olmalı — çünkü workflow .md path'leri Faz 12.1'den sonra `.kortext/` yapısına göre güncellenecek. Eşzamanlı yazılır ama merge sırası önemli.
