          Rate-limit signup endpoint to prevent abuse
          L-001
        

        
Discovered during B-203 analysis. Brute-force signup creates spam accounts. Add 5 req/min limit per IP + CAPTCHA after 3 failures.

        

          From: B-203 · devops-engineerApplied
          
        

        

          
Full Context

          
During investigation of B-203 (password reset email spam issue), devops-engineer noticed the /auth/signup endpoint has no rate limiting. Combined with no CAPTCHA, automated sign-up scripts can create thousands of spam accounts per hour, polluting the database and email queue. This also creates reputation risk with email providers.

          

            
Action items:

            

              
→ 10 req/min limit per IP on /auth/signup (T-109, pending approval)

              
→ CAPTCHA challenge after 3 failures from same IP

              
→ Account velocity check: flag accounts with no activity after 24h

            

          

        

      


      

        

          
          Stripe webhooks require idempotency keys
          L-002
        

        
Without idempotency keys, retry storms on webhook failures cause duplicate billing events. Always set Stripe-Idempotency-Key header.

        

          From: B-202 · backend-developer
          
        

        

          
Full Context

          
B-202 surfaced in staging: Stripe retries a failed webhook up to 3 times. Without an idempotency key on the handler, each retry creates a new subscription event in our database — customers were billed 3× in the test. Fix: derive idempotency key from Stripe event ID. Applied in commits to services/webhook.ts.

        

      


      

        

          
          Email deliverability: SPF + DKIM + DMARC all required
          L-003
        

        
Password reset emails going to spam was caused by missing DMARC record. All three DNS records are needed together.

        

          From: B-203 · devops-engineer
          
        

        

          
Full Context

          
SPF alone authenticates the sending server. DKIM proves message integrity. Without DMARC, Gmail's filtering treats the email as suspicious even if SPF+DKIM pass — because there's no policy telling receivers what to do with failed messages. All three must be in DNS. DMARC policy: p=quarantine initially, move to p=reject after monitoring for 7 days.
